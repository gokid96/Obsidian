Event-Driven Architecture는 **서비스 간 이벤트로 통신하는 방식**

---

## 기존 방식 vs Event-Driven

**기존 (직접 호출):** order-service가 다른 서비스를 직접 알고 호출

**Event-Driven:** order-service가 이벤트만 발행. 누가 구독하든 신경 안 씀

```
order-service ──이벤트──→ Kafka ──→ stock-service
                              ──→ notification-service
                              ──→ analytics-service
```

---

## 장단점

| 장점                     | 단점         |
| ---------------------- | ---------- |
| 느슨한 결합                 | 디버깅 어려움    |
| 확장 용이 (Consumer 추가 쉬움) | 순서 보장 어려움  |
| 장애 격리                  | 트랜잭션 관리 복잡 |
| 비동기 처리                 |            |

---

## 언제 무엇을?

| 동기 (REST) | 비동기 (Event-Driven) |
|-------------|----------------------|
| 즉시 응답 필요 | 응답 안 기다려도 됨 |
| 결과 확인 필수 | 여러 서비스가 반응 |
| 결제 요청 | 알림, 통계, 로그 |

**실무에선 섞어서 사용:**
```
주문 → 결제 (REST, 결과 필요)
    → 알림/재고/통계 (Kafka, 비동기)
```

---

## 구현 방식

**Spring Event** - 같은 JVM 내부, 서버 죽으면 유실
**Kafka** - 외부 브로커, 이벤트 보존/재처리 가능

---

## MSA와의 관계

| | MSA | Event-Driven |
|--|-----|--------------|
| 관심사 | 서비스 분리 | 서비스 간 통신 |
| 필수 관계 | ❌ | ❌ |

MSA 없이 Event-Driven 가능, MSA에서 REST만 사용도 가능.
