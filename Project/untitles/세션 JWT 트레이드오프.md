**핵심 차이: 상태 저장 위치**
- 세션: 서버가 상태를 저장
- JWT: 클라이언트가 상태를 저장

---

**세션 방식**
현재는 프레임워크가 쿠키 발급, 세션 저장, 요청마다 자동 조회를 해줌
이게 안되었을땐 세션 트래킹을썻음

> [!세션 트래킹]
> 1. 서버가 Set-Cookie: JSESSIONID=abc123 헤더로 응답 
> 2. 브라우저가 자동으로 쿠키 저장 
> 3. 이후 요청마다 Cookie: JSESSIONID=abc123 자동 전송 
> 4. 서버가 이걸로 세션 조회



장점
- 즉시 무효화 가능 (로그아웃, 강제 탈퇴 등)
- 서버가 모든 통제권 보유
- 토큰 탈취돼도 서버에서 삭제하면 끝

단점
- 매 요청마다 저장소(Redis, DB) 조회 필요
- 서버 여러 대면 세션 공유 문제 (sticky session 또는 중앙 저장소 필요)
- 수평 확장 시 복잡도 증가

---

**JWT 방식**

장점
- Stateless라 서버 확장 용이
- DB/Redis 조회 없이 토큰 자체로 검증 가능
- MSA 환경에서 서비스 간 인증 전달이 쉬움

단점
- 발급된 토큰 즉시 무효화 불가 (만료까지 유효)
- 토큰 크기가 큼 (페이로드에 정보 담으니까)
- 탈취당하면 만료 전까지 막을 방법이 마땅치 않음

---

**현실적인 트레이드오프**

|           | 세션  | JWT (순수) | JWT + Redis |
| --------- | --- | -------- | ----------- |
| 즉시 무효화    | O   | X        | O           |
| Stateless | X   | O        | △           |
| 확장성       | △   | O        | O           |
| 구현 복잡도    | 낮음  | 낮음       | 중간          |
결국 JWT를 쓰면서 보안을 챙기려고 Redis 블랙리스트나 Refresh Token 저장을 추가하면, 세션 방식과 비슷해지는 아이러니가 있습니다.

**선택 기준**

- 단일 서버, 보안 중요 → 세션
- MSA, 서버 간 인증 전파 필요 → JWT
- 둘 다 필요 → JWT + Redis (현실에서 가장 흔함)
