## 핵심 원칙
```
프롬프트 엔지니어링 → RAG → 파인튜닝
(비용 낮은 순서대로 시도)
```

## 문제별 해결책

| 문제 | 1차 시도 | 2차 시도 |
|------|----------|----------|
| 답변 형식이 안 맞음 | 프롬프트 개선 | Few-shot 예시 |
| 최신 정보 필요 | RAG | - |
| 내부 문서 기반 답변 | RAG | - |
| 특정 도메인 지식 부족 | RAG | 파인튜닝 |
| 할루시네이션 | RAG + 프롬프트 | - |
| 모델 톤/스타일 변경 | 프롬프트 | 파인튜닝 |
| 특정 태스크 성능 향상 | 프롬프트 | 파인튜닝 |

## 프롬프트 엔지니어링
**언제**: 거의 모든 경우 먼저 시도

**방법**:
- 명확한 지시사항
- 출력 형식 지정
- Few-shot 예시
- Chain-of-Thought

**한계**:
- 컨텍스트 길이 제한
- 새로운 지식 추가 불가

## RAG
**언제**:
- 최신 정보 필요
- 내부/독점 데이터 활용
- 출처 명시 필요
- 할루시네이션 줄이기

**장점**:
- 실시간 정보 반영
- 데이터 업데이트 쉬움
- 파인튜닝보다 저렴

**한계**:
- 검색 품질에 의존
- 모델 행동 자체는 안 바뀜

## 파인튜닝
**언제**:
- 특정 도메인 전문성 필요
- 일관된 스타일/톤 필요
- RAG로 해결 안 될 때
- 응답 시간 중요 (RAG 검색 시간 없앰)

**고려사항**:
- 데이터 준비 비용
- 학습 비용
- 유지보수 (모델 업데이트 시 재학습)

## 의사결정 플로우
```
Q: 새로운 지식이 필요한가?
├─ Yes → RAG
└─ No
    Q: 모델 행동/스타일을 바꿔야 하나?
    ├─ Yes → 파인튜닝 (프롬프트 먼저 시도)
    └─ No → 프롬프트 엔지니어링
```

## 비용 비교 (대략)

| 방법 | 초기 비용 | 운영 비용 |
|------|-----------|-----------|
| 프롬프트 | 없음 | 토큰 비용만 |
| RAG | 벡터DB 구축 | 검색 + 토큰 비용 |
| 파인튜닝 | 학습 비용 | 모델 호스팅 또는 API |
